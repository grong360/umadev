# 交付用上下文工程（商业级必读）

> 在长程交付里，**上下文是有限且会贬值的预算**，不是越塞越好。把整个代码库、整段历史、所有文档一股脑塞进去，换来的不是更聪明，而是**上下文腐化**（关键信息被淹没）、**上下文污染**（错误/过时信息被反复引用放大）、**上下文过载**（预算耗尽、注意力发散）。
> 这份规范把"喂上下文"升级成可执行的**工程纪律**：指令写在**恰当高度**、信息**即时检索**而非预先堆砌、压实（compaction）必须**保真**（决策/改动文件清单/测试命令一个不能丢）、用**外置便笺记忆**减负、子代理**上下文隔离**只回小摘要。它和 `ai/ai-agent-memory-context-management`（产品侧记忆分层）、`agentic-delivery/01-standards/self-improving-memory-and-regression-sets`（跨次记忆演进）互补：本规范回答的是"一次长程交付里，上下文该怎么管才不贬值"。

## 1. 上下文是预算：三种贬值与对策

| 贬值形态 | 表现 | 对策 |
|---|---|---|
| 上下文腐化（rot） | 关键信息被海量无关内容淹没，注意力发散 | 只放当前步骤所需，按相关性裁剪 |
| 上下文污染（poisoning） | 错误/过时/幻觉信息进了上下文并被反复引用放大 | 写入前校验来源、可追溯；发现污染即清并回退 |
| 上下文过载（overload） | 预算耗尽、长尾信息挤占决策所需空间 | 即时检索 + 压实 + 外置记忆，控制驻留量 |

原则：**最大化每一个 token 的信噪比**，而不是最大化塞进去的量。

## 2. 恰当高度的指令（right-altitude）

指令既不能太低（把每一步都写死成脆弱的硬编码流程），也不能太高（"做好它"这种无法执行的空话）：

- **太低**：穷举式 if-this-then-that 脚本，环境一变就脆，且占满预算。
- **太高**：只给愿景不给可判定边界，产出发散、无法验收。
- **恰当**：给**目标 + 约束 + 验收标准 + 可用工具**，把"怎么做"的空间留给执行、把"做到什么算对"钉死。结构化需求与验收来自 `experts/product-manager/requirements-engineering-ears`。

## 3. 即时检索（JIT）优于预先堆砌

- **预先堆砌的代价**：把整个代码库/全部文档塞进上下文 → 信噪比塌陷、预算瞬间见底。
- **即时检索**：按当前步骤的需要**临时取**最相关的切片（相关代码、相关契约、相关知识），用完即让其退出驻留。
- **保留指针而非全文**：上下文里留"在哪能取到"（文件路径、检索入口）而非把全文常驻，需要时再拉。
- **代码库切片**：用仓库地图/符号大纲给"够用的结构"而非全量源码，控制驻留体积。

## 4. 压实（compaction）必须保真

长会话逼近预算时要压实，但压实是**有损操作**，必须保住交付不可丢的事实：

| 压实后必须保留 | 为什么 |
|---|---|
| 已做的关键决策与其理由 | 否则后续会推翻或重做已定方案 |
| **已修改文件清单** | 否则丢失"改了哪些地方"，回归与收尾失准 |
| **测试/构建/验证命令** | 否则无法重新验证，验收断链 |
| 未决问题 / 待办 / 已知坑 | 否则同一个坑再踩一遍 |
| 当前计划与下一步 | 否则上下文一压就迷失目标 |

压实规则：宁可丢掉寒暄与探索弯路，**绝不丢决策、改动文件清单、验证命令**。压实是摘要不是清空。

## 5. 外置便笺记忆（scratchpad）

- 把超出当前必须驻留的信息**写到上下文之外**（便笺/文件/工作区产物），需要时再读回，给上下文减负。
- 计划、决策记录、改动清单、验证结果落成**可寻址的外部产物**，而不是只活在易逝的会话里——这也让交付可审计、可恢复。
- 这与跨次的自演进记忆衔接（见 `agentic-delivery/01-standards/self-improving-memory-and-regression-sets`）：便笺是本次的工作记忆，演进记忆是跨次的长期沉淀。

## 6. 子代理上下文隔离（只回小摘要）

把发散的子任务（深度检索、批量扫描、探索性分析）交给**独立上下文**的子代理处理，主线只接收**精炼摘要**：

- **隔离**：子代理在自己的上下文里展开海量中间过程，主线不被其噪声污染。
- **小摘要回传**：子代理只返回主线决策真正需要的结论 + 证据指针，而非全部原始过程。
- **预算守恒**：主线上下文始终保持高信噪比，长程任务不被单个子任务撑爆。

## 7. 接入交付流程

- **每步开始**：JIT 取该步所需切片，指令给在恰当高度。
- **逼近预算**：压实，保住决策/改动文件清单/验证命令。
- **发散子任务**：丢给隔离子代理，只收小摘要。
- **持续**：关键状态外置成可寻址产物，污染即清并回退。

## 8. 反模式（出现即不合格）

1. **把整个代码库/全历史预先塞满**：信噪比塌陷、预算见底、决策被淹。
2. **指令太低或太高**：脆弱硬编码流程，或"做好它"式空话，无法执行/无法验收。
3. **压实丢掉决策/改动文件清单/验证命令**：回归失准、验收断链、方案被重做。
4. **放任上下文污染**：错误/过时信息反复引用被放大，无人清除。
5. **全文常驻不留指针**：该退出驻留的全文一直占着预算。
6. **子任务不隔离**：探索噪声直接灌进主线，把主上下文撑爆。
7. **关键状态只活在会话里**：一压实/一中断就全丢，不可恢复、不可审计。

## 9. 最低交付 checklist

- [ ] 指令写在恰当高度：给目标+约束+验收+工具，不写死脆弱流程、不留空话。
- [ ] 按步骤即时检索相关切片，用仓库地图/指针替代全量常驻。
- [ ] 压实保真：决策、已修改文件清单、测试/构建/验证命令一律保留。
- [ ] 关键状态（计划/决策/改动清单/验证结果）外置为可寻址产物，可恢复可审计。
- [ ] 发散子任务交隔离子代理，主线只接收精炼摘要 + 证据指针。
- [ ] 上下文污染可检测：写入前校验来源，发现即清并回退。
- [ ] 全程守住信噪比，预算用于决策所需信息而非长尾堆砌。
