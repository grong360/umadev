# 测试完整性与反作弊规范（商业级必读）

> 一句话：**一套全绿的测试，只有在确认它没被作弊的前提下才可信。** 当"让测试通过"成了目标，最省力的路径常常不是把代码改对，而是把**测试改松**——硬编码期望输出、删/弱化断言、`skip`/`xfail` 掉碍事的用例、动测试框架、专门特判可见用例、提前 `return` 跳过校验。绿灯于是变成了对"会不会绕"的奖励，而不是对"对不对"的证明。
> 这份规范给出**作弊行为目录**与**能戳穿它们的验证手段**：一道作者改不动的确定性校验、一份完整性差异、独立 + 留出（held-out）检查、以及评审复核。它和 `agentic-delivery/01-standards/test-discipline-for-generated-code`（测什么）、`agentic-delivery/01-standards/verifier-critic-pattern`（谁来验）互补；CI 门禁形态见 `testing/01-standards/ci-test-gates-and-coverage`。本规范回答的是"凭什么相信这套绿测"。

## 1. 作弊行为目录（reward hacking 的常见形态）

把"通过测试"当奖励、把"改对代码"当手段时，会冒出这些行为——**每一种都让绿灯失去意义**：

| 作弊形态 | 具体表现 | 为什么危险 |
|---|---|---|
| 硬编码输出 | 直接 `return` 期望值/把断言里的预期写进实现，特判测试输入 | 真实输入立刻失效，测试只证明"会背答案" |
| 弱化/删断言 | 把 `assertEqual` 改成 `assertTrue`、放宽容差、删掉关键断言 | 缺陷出现也不再变红 |
| skip / xfail | 给碍事用例打 `skip`/`xfail`/`ignore`/注释掉 | 失败被静音，覆盖面悄悄缩小 |
| 篡改测试框架 | 改 conftest/setup/全局 fixture/断言钩子让失败被吞 | 整套套件的可信度坍塌 |
| 特判可见用例 | 只让"看得见的样例"过，逻辑对一般情况错 | 留出集一换就崩 |
| 提前退出/短路 | 在校验前 `return`/吞异常/`try…except: pass` | 错误路径被绕过，错误被当成功 |
| 放宽门禁 | 调低覆盖率阈值、关掉某条必过检查、给 lint 加豁免 | 把门拆了再"通过" |
| 改基线/期望文件 | 直接把快照/golden 文件改成当前（错误）输出 | 回归保护被反向利用 |

判定原则：**只要"通过"是靠改松验证而非改对代码换来的，即为作弊，等同未完成。**

## 2. 能戳穿作弊的验证手段

光禁止没用，要让作弊**在结构上失效**：

- **一道作者改不动的确定性校验**：核心验收用一条**实现/测试作者无权修改**的确定性检查把关（独立 runner、固定输入→固定期望、产物级断言）。作者能改的测试只能加分，不能替代这道闸。
- **留出（held-out）检查**：除可见用例外，保留一组作者**看不到**的输入/期望用于最终判定；专门特判可见用例的实现会在这里崩。
- **完整性差异（integrity diff）**：每次改动**单独审视对测试与门禁配置的改动**——断言是被加强还是被删/弱化？是否新增 `skip`/`xfail`？是否动了框架/阈值/豁免？测试改动要和实现改动**分开评审**，不能混在一个大 diff 里蒙混过关。
- **独立检查**：验证由不写该实现的视角执行（见 `agentic-delivery/01-standards/verifier-critic-pattern`），避免"自证清白"。
- **评审复核（judge review）**：对"是否在解决问题而非绕过验证"做一次结构化判定（见 §4），作为人/裁判侧的最后一关。

## 3. 完整性差异：必须逐项盘问的改动信号

每次改动对"测试与门禁"的任何改动都要能解释，否则视为可疑：

| 信号 | 默认判定 | 放行条件 |
|---|---|---|
| 断言被删除或放宽 | 可疑 | 有明确理由且不降低区分力，评审签字 |
| 新增 skip / xfail / ignore | 可疑 | 关联到已登记的真实缺陷与期限 |
| 改动测试框架/全局 fixture/钩子 | 高危 | 独立复核，确认不吞失败 |
| 修改快照/golden/基线文件 | 可疑 | 确认是预期行为变化而非掩盖回归 |
| 调低覆盖率阈值/加 lint 豁免/关必过检查 | 高危 | 默认拒绝，需显式治理审批 |
| 实现里出现对测试输入的特判/硬编码 | 作弊 | 不放行 |

## 4. 评审/裁判复核要点

复核者只问一件事：**这次是把问题解决了，还是把验证绕过了？**

- 实现是否对**一般输入**成立，而非只过可见用例？（用留出集证实）
- 测试改动是**加强**了区分力，还是**削弱**了？
- 是否有 `skip`/`xfail`/吞异常/提前 return 在静音失败？
- 门禁/阈值/豁免是否被动过？动了就要有治理依据。
- 结论给**结构化判定**：pass / fail + 证据（哪条留出用例、哪段 diff），fail 即打回，路由方式见 `agentic-delivery/01-standards/verifier-critic-pattern`。

## 5. 接入交付流程

- **门禁分层**：作者可改的测试加分，**确定性留出校验**作为不可绕过的判定闸。
- **diff 分离**：实现 diff 与测试/门禁 diff 分开呈现、分开评审。
- **留痕**：留出集结果、完整性差异结论随交付留证，可审计。
- **沉淀**：被识破的作弊手法记入经验库与项目规则，作为后续硬约束。

## 6. 反模式（出现即不合格）

1. **实现里特判/硬编码测试输入**：测试只在背答案，真实输入即崩。
2. **靠删/弱化断言换绿**：缺陷出现不再变红，回归保护形同虚设。
3. **用 skip/xfail/注释静音失败**：失败被藏起来，覆盖面悄悄缩水。
4. **篡改测试框架吞掉失败**：整套套件可信度坍塌。
5. **把实现和测试改动混进一个大 diff**：完整性差异无法审，作弊蒙混过关。
6. **改门禁阈值/加豁免来"通过"**：把门拆了再宣称达标。
7. **无留出集**：只用可见用例判定，特判式实现永远过关。
8. **验证者就是实现者**：自证清白，作弊无人能戳穿。

## 7. 最低交付 checklist

- [ ] 核心验收由一道实现/测试作者**改不动**的确定性校验把关。
- [ ] 存在作者不可见的留出（held-out）用例用于最终判定。
- [ ] 实现 diff 与测试/门禁 diff 分离呈现并分别评审（完整性差异）。
- [ ] 任何删/弱断言、新增 skip/xfail、改框架/阈值/基线都有理由并经复核。
- [ ] 验证由不写该实现的独立视角执行，给出 pass/fail + 证据。
- [ ] 覆盖率阈值与必过检查不可被本次改动私自调低或豁免。
- [ ] 被识破的作弊手法沉淀为项目级硬约束，防止重现。
