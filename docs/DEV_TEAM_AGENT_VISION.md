# UmaDev 定位:一支 AI 开发团队(Development Team Agent)

> 本文档确立 UmaDev 的**对外定位与发展方向**,取代旧的"项目总监 Agent(project-director Agent)"叙事。
> 旧叙事把"一个总监"当主角是错的 —— **真正的产品、真正打动人的,是那支团队。**

---

## 一句话定位

**UmaDev 是一支 AI 开发团队 —— 产品经理、架构师、UI/UX 设计师、前端、后端、QA、安全、DevOps 八个角色像真实团队一样分工协作,借你已登录的底座大脑(Claude Code / Codex / OpenCode),把一句需求做成能上线、能交付、能审计的商业级应用。**

- 你不是雇了一个总监,你是**雇了一整支开发团队**。
- 独立开发者、小团队,瞬间拥有一支完整、专业、有工程纪律的团队。
- 总监("技术负责人/协调者")只是团队里的一个协调角色,**不是头牌** —— 团队才是。

## 为什么是"团队",不是"总监"

- **更准**:架构本来就是团队架构(八个角色席位 + 协调者),旧叙事的"一个总监"反而遮蔽了真正的差异化。
- **更有钩子**:"一支 AI 开发团队帮你干活"比"一个项目总监"在情绪上强得多 —— 它直接回答了独立开发者最大的痛:我一个人扛不动产品/架构/前端/后端/测试/安全/运维所有维度。
- **更解释得通商业级**:商业级缺的那一段(需求拆清、契约对齐、真测试、不泄密、反 AI 模板、可交付证据),恰恰是**一个人/一个模型兜不住、需要一支分工团队**才能补上的。

## 团队成员(对外用"团队/角色"语言,不用"评审席位"这种内部黑话)

| 角色 | 干什么(真实团队该产出的) |
|---|---|
| 产品经理 | 把需求拆清、写 PRD、定验收 |
| 架构师 | 定分层、服务边界、API 契约、技术选型 |
| UI/UX 设计师 | 定设计系统(字体/令牌/组件/页面骨架)、反 AI 模板审美 |
| 前端工程师 | 按设计系统 + 契约真建前端、跑通运行时 |
| 后端工程师 | 建数据模型 + API + 业务逻辑,对齐契约 |
| QA | 真跑构建/测试、覆盖核对、运行时证明 |
| 安全工程师 | 语义攻击面审查(鉴权/越权/注入/密钥) |
| DevOps | 构建/发布/部署证明 |
| 技术负责人(协调者) | 路由意图、拆计划、调度团队、把关每道门、留证据 |

## 发展方向:让每个角色"真正干活",且贡献可见

旧实现偏"协调者驱动主会话干完、角色点评"。往**真实开发团队**发展 =

1. **每个角色真正干自己那摊活**(实干角色不止前端/后端 —— 设计师产出设计系统、架构师产出契约、QA 产出运行时证明……),而不只是事后评审。
2. **贡献可见**:用户看得见"现在是设计师在定设计系统""现在是后端在建 API""安全工程师发现了一个越权" —— 像看一支真团队在协作,而不是一个黑盒。
3. **保持已验证的健壮性(四条铁律不动)**:单写者(同一时刻只有主会话动文件)、评审角色只读分叉并行、角色之间不互聊(只通过黑板 + 结构化裁决)、团队规模随任务缩放。这套正是同时吃到并行红利又躲开多 agent 脆弱性的关键,**不能为了"更像团队"把它推翻**。
4. **确定性地板仍是裁判**:角色裁决只是建议,过门/返工/硬停由确定性信号(零代码硬门、构建测试退出码、契约、覆盖、有界 gap/stall)决定。模型不给自己打分。

> 具体的角色协作模型、实干/评审平衡、每个角色的可见产出,由 `docs/` 下的开发团队 Agent 架构研究产出细化(进行中),再落到实现。

## 什么不变(护城河)

固件(借脑不造脑)+ 六根支柱仍是核心,只是叙事的主角从"总监"换成"团队":
团队分工 · 确定性控环 · 借脑(驱动你已登录的底座)· 工程品味 + 代码库记忆(知识库 RAG + repo-map)· 治理即代码(实时拦截)· 自学习记忆。

## 落地蓝图(R1 架构研究结论,已落地化)

**关键发现:UmaDev 已 70% 是团队架构 —— 团队早就在数据模型里。** `plan_state.rs` 的 `PlanStep` 已带 `seat`(角色席位)+ `kind{Build,Review}` + `acceptance` + `depends_on`;`critics.rs` 有 8 席位 + `team_for_kind` 缩放 + 结构化 `RoleVerdict`;`director_loop.rs` 的 `drive_plan_steps`/`drive_build_step`(按席位逐步建造,"Wave 2")**已写好**。缺口在 `director.rs` header 明说的:**默认路径退役了按席位建造**(底座一轮内自己建多角色代码,UmaDev 只事后评审 = "Wave 1")—— 这正是它"像总监带评审"而非"像团队在建造"的根因。

### 健壮协作模型(Anthropic vs Cognition 的统一)
分清**读 vs 写**,两派就不矛盾了:**并行只读评审 + 串行单写建造 + 黑板交接**。
- 同一时刻只一个角色写(单写者,串行驱动同一连续上下文)—— Cognition 的"单线程线性 agent"。
- 并行 agent 一律只读、只答有界问题(评审角色 fork 只读,回结构化裁决)—— 安全地拿到 Anthropic 的并行红利。
- 角色之间绝不互聊,只通过黑板(`output/*.md` + 源码 + `.umadev/contracts/`)+ 结构化裁决交接 —— 这条是杀脆弱性的命门,团队扩大也不能破。
- 确定性地板(非任何 LLM 裁决)管循环终止。
- **结论:UmaDev 现有四条铁律就是这个健壮综合,不准放松;演进只动"实干"侧,不动"评审"侧。**
- 硬证据(AgentCoder,91.5% vs 86.8% pass@1):**写测试的角色 ≠ 写代码的角色**(同一 agent 出的测试有偏),所以 **QA 要真写测试,不只评审**。

### 每个角色:从"评审"升级为"产出自己的产物"
| 席位 | 真正产出(写到黑板)| 机器验收 | 下游消费者 | 可见为 |
|---|---|---|---|---|
| 产品经理 | `output/*-prd.md`(用户故事 + EARS 验收 + FR)| coverage FR→任务 | 架构师/QA/覆盖门 | "PRD 就绪 · 14 需求"卡片 |
| 架构师 | `*-architecture.md` + **`.umadev/contracts/openapi.*`** + 数据模型 | Contract(契约前后端对照)| 前端/后端(契约即交接)| "契约锁定 · 9 端点" |
| 设计师 | `*-uiux.md` + **`design-tokens.{json,css}`** + 组件清单 + 页面骨架 | **新增 DesignTokensPresent**(令牌存在 + 治理拦 emoji/硬编码色)| 前端(必须导入令牌)| "设计系统 24 令牌 11 组件" |
| 前端 | 导入令牌 + 调契约 URL 的真实组件/页面 | SourcePresent→BuildTest→Contract | QA/运行时证明 | 实时 PlanStepStatus 泳道 |
| 后端 | 对齐契约的端点 + 迁移 | Contract + BuildTest | QA/安全 | 实时泳道 |
| QA | **先写测试** + `runtime-proof.json` | BuildTest + 路由探测 | 地板(放行合并)| "QA 31 测试 路由 200" |
| 安全 | 威胁模型 + SAST | security 基线无新高危 | PR 门 | "安全 0 新高危" |
| DevOps | CI + `deploy-proof.json` | deploy 证明 | 交付 | "已部署 · 预览 URL" |

### 三个要新增的产物(让角色"真实"的最高性价比升级)
1. **design-tokens 成一等产物** + `DesignTokensPresent` 验收 —— 设计师从"评审"变"真出设计系统",直接接现有反 AI-slop 治理。
2. **QA 先写测试**(AgentCoder 偏差结论),不只核覆盖。
3. **契约优先硬排序** —— 架构师 OpenAPI 成所有前后端 step 的 `depends_on` 前驱(已有 umadev-contract,只需进 DAG)。

### 可见性(信任层 —— 让它"像真团队")
团队花名册面板(席位为命名队友 + 实时状态 idle/working/reviewing/blocked/done)· 产物卡片(完成即给可点开真实产物的卡,文件存在+过验收才显示,反剧场)· 交接时间线("架构师→发布契约→前后端接手")· 每席位裁决 chip · `/team` 命令(对标 `/plan`)· 新事件 `SeatConvened`/`SeatStatus`/`DeliverableProduced`。

### 反剧场铁律("每个席位都要配得上"——和 fail-open 一样硬)
1. 无产物不设席位:一个角色被召集,当且仅当它有(a)具体产物(b)机器验收(c)下游消费者。
2. 交接=产物,绝不=旁白:角色叙述"我设计好了"但没 design-tokens 文件,DAG 不前进。
3. 专业化必须改变产出:写代码的 ≠ 判代码的(去偏),无增益处不强分。

### 迁移波次(每一波都不碰四条铁律 + 确定性地板 + fail-open)
- **Wave A(重排序)**:把默认构建翻成按席位 `drive_plan_steps`;单写者+地板+fail-open 不变,Wave-1 单轮做兜底。角色开始"建造"而非只"评审"。
- **Wave B(产物)**:design-tokens + 契约优先 DAG + QA 先写测试 的 AcceptanceSpec。
- **Wave C(可见)**:花名册面板 + 产物卡 + 交接时间线 + `/team` + constitution 文件(把 L0 固件不可破原则落成用户可见可编辑的团队章程)。
- **Wave D(打磨)**:每席位 trace/usage、可续接席位、按复杂度命名的团队规模。

> 代码锚点:`plan_state.rs`(seat/kind/acceptance 已在模型)· `critics.rs`(Seat/RoleVerdict/team_for_kind)· `director.rs`(退役了席位默认的那段 header——要反转的地方)· `director_loop.rs`(drive_plan_steps 已建,提为默认)· `events.rs`(已有 IntentDecided/PlanPosted/PlanStepStatus/CriticVerdict)· umadev-contract / runtime_proof / security / deploy / coverage(现成的角色产物机器)。
