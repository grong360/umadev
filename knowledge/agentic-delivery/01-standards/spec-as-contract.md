# 规格即契约（商业级必读）

> 一份"含糊的需求描述"换来的是含糊的实现。当交付由生成式执行时，规格不再只是给人读的说明，而是**给执行者的契约**——它必须**自包含**到：点名要改的文件与接口、明说**不做什么**、钉死版本、把已知的坑**写进去**、并以一条**端到端验证步骤**收尾。规格里没说清的，执行者就会自己猜，而猜出来的往往就是返工。
> 这份规范定义"能直接执行的规格"长什么样，是 `experts/product-manager/requirements-engineering-ears`（把每条需求写成可验收句式）的工程化延伸——EARS 解决"单条需求怎么写得可验"，本规范解决"整份规格怎么自包含到可被直接执行、且随实现保持鲜活"。契约接口细节见 `experts/architect/contract-first-api-design`。本规范回答的是"一份规格要含到什么程度，执行者才不用猜"。

## 1. 为什么规格要当契约写

- **消除猜测**：执行者不在你脑子里。凡是规格没钉死的（哪个文件、什么接口、哪个版本、什么不做），都会被自由发挥，发挥的结果常与意图不符。
- **可验收**：契约式规格自带"做到什么算对"，验收有据可依（呼应 `agentic-delivery/01-standards/verifier-critic-pattern` 的"只看 diff + 验收标准"）。
- **可并行**：点名了文件与接口、明确了范围，多个工作面可并行而不互相踩。
- **抗返工**：把已知坑写进规格，是把"事后踩坑"换成"事前规避"，最便宜的质量。

## 2. 自包含规格的必含要素

一份能被直接执行的规格，必须当场答全下面这些，缺一项就会变成猜：

| 要素 | 要求 |
|---|---|
| 目标与背景 | 要解决什么问题、为什么、成功长什么样 |
| **点名文件/模块** | 明确要改/要新增哪些文件、模块、目录，而非"在相关地方" |
| **点名接口** | 涉及的函数/方法/API/契约签名与数据形状，按真实定义写 |
| 验收标准 | 每条需求用可验收句式（见 EARS），含边界/错误/权限/并发 |
| **明确不做（out-of-scope）** | 显式列出本次**不**覆盖什么，杜绝镀金与范围蔓延 |
| **版本锚定** | 钉死框架/库/运行时的实际版本，按该版本 API 写 |
| **内嵌已知坑** | 把这块历史上踩过、易错的点写进规格作为硬约束 |
| 依赖与影响面 | 上下游依赖、数据/契约影响、迁移注意 |
| **端到端验证步骤** | 收尾一条"怎么从头到尾证明它真的可用"的可执行验证 |

## 3. 三个最常被略过、最该钉死的

- **明确不做（out-of-scope）**：不写清楚不做什么，执行者就会顺手加未被要求的特性——这既是浪费也是风险（见 `agentic-delivery/01-standards/verifier-critic-pattern` 的过度设计护栏）。每份规格都要有一节"本次不包含"。
- **版本锚定**：不锚版本，执行者会按"通用印象"写 API，版本不符即幻觉签名（见 `agentic-delivery/01-standards/generated-code-failure-modes`）。先确认已安装的实际版本，再按它写。
- **内嵌已知坑**：把项目记忆里这块的高频坑（见 `agentic-delivery/01-standards/self-improving-memory-and-regression-sets`）直接写进规格，让执行者事前避开，而不是又踩一遍再被回归集抓。

## 4. 端到端验证步骤：规格的收尾即验收

规格不能停在"实现完成"，要停在"怎么证明它对"：

- **可执行**：给出从头到尾跑一遍核心流的具体步骤/命令（构建、启动、关键路径走查、断言），而不是"测一下"。
- **覆盖关键路径**：端到端验证至少覆盖核心成功流 + 一条关键错误/边界流。
- **与回归衔接**：端到端验证里暴露的问题沉淀成持久回归用例（见 `agentic-delivery/01-standards/self-improving-memory-and-regression-sets`）。
- **作为完成定义**：端到端验证通过 + 验收标准满足 + 回归率为零，才算这份规格交付完成。

## 5. 活规格（living-spec）维护

规格不是写完即弃的一次性文档，它随实现保持鲜活：

- **同步更新**：实现中发现规格与现实冲突（接口变了、范围调整、新坑），**先改规格再改代码**，不让规格与实现失同步。
- **决策留痕**：规格里关键决策与其理由保留，避免后续推翻重做（与压实保真一脉相承，见 `agentic-delivery/01-standards/context-engineering-for-delivery`）。
- **过时即修**：被证伪的约束显式修正，不放任过时条目误导后续。
- **可寻址**：规格作为可寻址的交付产物存在（而非只活在会话里），可审计、可恢复。

## 6. 接入交付流程

- **动工前**：产出自包含规格（点名文件/接口、钉死范围与版本、内嵌坑、端到端验证步骤）。
- **执行中**：规格与实现冲突时先改规格；执行者只按规格 + 验收标准做。
- **收尾**：跑端到端验证步骤，达标 + 回归率零才判完成。
- **沉淀**：验证暴露的问题进回归集，新坑回写进规格与记忆。

## 7. 反模式（出现即不合格）

1. **含糊需求当规格**："优化一下登录"这种，逼执行者全程猜。
2. **不点名文件/接口**："在相关地方改"，执行者改错位置或漏改。
3. **不写 out-of-scope**：放任镀金与范围蔓延，做了一堆没被要求的活。
4. **不锚版本**：按通用印象写 API，版本不符即幻觉签名。
5. **已知坑不写进规格**：让执行者把团队踩过的坑再踩一遍。
6. **规格停在"实现完成"**：没有端到端验证步骤，无法证明真的对。
7. **规格写完即弃**：实现改了规格不改，二者失同步、误导后续。

## 8. 最低交付 checklist

- [ ] 规格自包含：点名要改/新增的文件与模块，写清涉及的接口签名与数据形状。
- [ ] 每条需求用可验收句式（EARS），含边界/错误/权限/并发。
- [ ] 显式列出"本次不做（out-of-scope）"，杜绝镀金与范围蔓延。
- [ ] 钉死框架/库/运行时实际版本，按该版本 API 写。
- [ ] 把该模块已知高频坑内嵌进规格作为硬约束。
- [ ] 规格以一条可执行的端到端验证步骤收尾（覆盖核心成功流 + 关键错误流）。
- [ ] 活规格维护：实现冲突先改规格、决策留痕、过时即修，作为可寻址产物可审计。
