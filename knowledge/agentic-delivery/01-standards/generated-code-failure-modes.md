# 生成式代码失效模式目录（商业级必读）

> 生成式代码在演示里"能跑"，在生产里翻车，几乎总是因为同一批**可预测的失效模式**：只覆盖 happy path、把不在测试里的行为悄悄写错、引用不存在的依赖/接口、缺少系统约束（权限/限额/网络策略）、对性能毫无考虑、import 缺失却静默失败。这些不是随机 bug，是这一类产出的**结构性偏差**——可以逐条预判、逐条防住。
> 这份规范把每一类失效配上**唯一能治它的纪律**：从"愿景式提醒"升级成"出现即不合格 + 配套防线"。测试侧的定向与回归纪律见 `agentic-delivery/01-standards/test-discipline-for-generated-code`，验证者解耦见 `agentic-delivery/01-standards/verifier-critic-pattern`，错误模型见 `backend/01-standards/error-handling-taxonomy`，供应链/依赖核验见 `security/01-standards/supply-chain-security`。本规范回答的是"生成式代码会怎么坏、对应要立哪条死规矩"。

## 1. 失效模式 × 纠偏纪律（核心对照表）

| 失效模式 | 典型表现 | 危险等级 | 纠偏纪律（唯一治法） |
|---|---|---|---|
| Happy-path / 错误处理偏差 | 只写成功路径，错误/边界/超时分支缺失或仅 `catch` 后吞掉 | 高 | 按错误分类法穷举失败路径并补错误分支用例 |
| **沉默逻辑错误（最致命）** | 不在测试覆盖内的行为被悄悄写错，编译通过、跑得动、结果错 | **极高** | 用影响图把测试火力压到未覆盖行为 + 变异定向加固 |
| 幻觉依赖/接口 | import 不存在的包、调用不存在的方法/参数、编造 API 形状 | 高 | 依赖与接口**存在性核验**（见 §3） |
| 缺失系统上下文 | 漏权限(RBAC)、限额/配额、速率限制、网络/防火墙策略、多租户隔离 | 高 | 把系统约束作为显式输入清单注入并逐项验收 |
| 性能无考量 | N+1 查询、循环内 IO、无分页/无索引、全量加载、无超时 | 中高 | 对热点路径做复杂度与容量审查 |
| 静默缺失 import / 未用结果 | 漏 import 在动态语言里运行时才炸、函数返回值未被使用 | 中 | 静态检查 + 构建/lint 必过门禁 |

判定原则：以上每一类**出现即不合格**；尤其"沉默逻辑错误"——它不报错、不崩溃、只是悄悄算错，是所有失效里最贵的一类。

## 2. 沉默逻辑错误：为什么最致命，怎么防

- **本质**：代码能编译、能运行、对演示输入看起来对，但对**没有被测试触及**的行为给出错误结果——没有任何信号告诉你它错了。
- **防线**：
  - 用代码↔测试影响图（见 `agentic-delivery/01-standards/test-discipline-for-generated-code`）把测试压到**未被覆盖的关键行为**上，而不是只测改了的那几行。
  - 对核心逻辑做变异定向加固：注入你最怕的算错方式，要求有测试在它出现时变红。
  - 用独立验证者只看 diff + 验收标准复核（见 `agentic-delivery/01-standards/verifier-critic-pattern`），抓"看起来对、其实需求没满足"的偏差。
- **反例**：把货币四舍五入、时区换算、权限取反、分页 off-by-one 当成"反正测试绿了"——这些正是沉默逻辑错误的高发区。

## 3. 幻觉依赖/接口：存在性核验是硬门

生成式产出最常编造**不存在的包和方法**。把"存在性"当作可阻断门：

- **包存在性**：每个新增依赖必须在锁定的来源里**真实存在且版本可解析**，不接受"看起来很合理"的包名（错别字/抢注风险，见 `security/01-standards/supply-chain-security`）。
- **接口存在性**：调用的方法/参数/返回形状必须对照真实接口文档或类型定义核验，禁止凭记忆编 API 签名。
- **版本锚定**：先确认已安装框架/库的**实际版本**，按该版本的 API 写代码，不按"通用印象"写（版本不符的签名是幻觉的重灾区）。
- **构建即验证**：解析/编译/类型检查作为必过门，幻觉 import 在这里就该红。

## 4. 缺失系统上下文：把隐性约束变成显式清单

生成式产出默认不知道你的系统边界，必须把约束**喂进去并逐项验收**：

| 约束类别 | 必须显式确认 |
|---|---|
| 鉴权与授权 | 谁能访问、RBAC/owner 校验、未登录/越权分支 |
| 限额与配额 | 速率限制、配额、分页上限、批量上限 |
| 网络与隔离 | 网络策略/防火墙、多租户数据隔离、出网白名单 |
| 数据与合规 | 敏感数据处理、加密、保留与删除策略 |
| 幂等与并发 | 重试安全、幂等键、并发写冲突 |

这些不是"有空补"，是验收项；漏一项即视为缺失系统上下文失效。

## 5. 性能盲区与静默缺失

- **性能**：对热点路径审查 N+1、循环内 IO、缺分页/索引、全量加载、缺超时；至少给出复杂度量级判断，深度见 `development/performance-capacity-complete`。
- **静默缺失**：漏 import、未使用返回值、未处理 Promise/错误，靠静态检查 + 构建/lint 门拦住（见 `testing/01-standards/ci-test-gates-and-coverage`），不留到运行时。

## 6. 接入交付流程

- **生成前**：把系统约束清单 + 已安装版本作为显式输入。
- **生成后**：构建/类型检查/lint 必过拦截幻觉与静默缺失；依赖/接口过存在性核验。
- **验证**：影响图 + 变异加固防沉默逻辑错误；独立验证者复核需求满足。
- **沉淀**：每类被抓到的失效记入项目级回归集与经验库，防止重现。

## 7. 反模式（出现即不合格）

1. **只交付 happy path**：错误/边界/超时分支缺失或被吞掉。
2. **放任沉默逻辑错误**："测试绿了"就签字，未覆盖行为悄悄算错无人发现。
3. **引用幻觉依赖/接口**：装不存在的包、调编造的方法签名。
4. **无视系统约束**：漏权限/限额/网络策略/租户隔离直接上。
5. **对性能零考量**：N+1、循环内 IO、无分页/索引照单全收。
6. **靠运行时才暴露静默缺失**：漏 import/未处理错误不被构建门拦住。
7. **凭印象写 API 签名**：不核版本、不查接口，编造参数与返回形状。

## 8. 最低交付 checklist

- [ ] 每类失效模式都有对应防线，且"出现即不合格"写进验收。
- [ ] 错误/边界/超时分支齐全并各有用例，不只覆盖 happy path。
- [ ] 未被测试覆盖的关键行为有定向测试 + 核心逻辑变异加固，防沉默逻辑错误。
- [ ] 新增依赖与调用接口过存在性核验，按已安装版本写、构建即验证。
- [ ] 权限/限额/网络/隔离/合规作为显式约束清单逐项验收。
- [ ] 热点路径过性能审查（N+1/分页/索引/超时）。
- [ ] 漏 import/未处理错误被静态检查 + 构建/lint 门拦住，不留到运行时。
